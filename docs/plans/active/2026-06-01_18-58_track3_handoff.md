## THE DECISION — pick a Track 3 entry point

Five candidates (roadmap order). Effort is rough; "touches" names the real files.

### 1. True Lewis–Syrgkanis recursive dynamic g-estimation  ← headline methodology gap
- **What:** a new, honestly-named estimator (recursive/sequential g-estimation with
  period-specific `theta_t`) — the exact thing `CLAUDE.md` says `TemporalPLRDML` is **not**.
- **Touches:** new `src/dml/dynamic_g_estimation.py` (name TBD), a dynamic-treatment DGP in
  `src/validation/`, tests, a runnable `examples/` script, and the book section that
  currently only *describes* it. Do **not** revive old `DynamicDML` aliases (CLAUDE.md rule).
- **Effort:** Large (research + interface + validation-against-DGP + HAC inference + docs).
- **Sub-decisions to resolve first (use /exploring-options):** (a) full recursive
  g-estimation vs a clearly-labeled simplified variant; (b) native implementation vs wrap
  EconML `DynamicDML`; (c) scope of period-specific `theta_t`; (d) validation DGP; (e) class
  name. **First step:** Lewis–Syrgkanis (2021) re-read + interface sketch + a validation DGP.

### 2. Package namespace migration off `src.*`
- **What:** promote the staged `src.*` imports to a real public package name — the cleanup
  `CLAUDE.md` defers ("keep imports under the staged `src.*` namespace until the
  documentation/API cleanup has passed").
- **Touches:** `pyproject.toml` (name/packages/package-dir), **every** import in
  `src/ test/ examples/`, Sphinx docs, CI. Mechanical but wide.
- **Effort:** Medium-large (codemod-able; breakage risk is the whole import surface).
- **Sequencing note:** cheapest to do **before** #1 so new code isn't renamed later.
- **Sub-decisions:** (a) package name; (b) keep `src/`-layout vs flat; (c) any transitional
  shims. **First step:** choose name → `pyproject.toml` → codemod imports → green test run.

### 3. EconML heterogeneity examples as runnable, tested features
- **What:** turn Ch9's EconML prose/code (CausalForestDML / LinearDML / BLP / policy tree)
  into actually-runnable, tested examples.
- **Touches:** `examples/`, `test/`, an `econml` extra in `pyproject.toml`.
- **Effort:** Small-medium — **cheapest win**, complements the already-shipped Ch9.
- **Sub-decisions:** (a) which estimators; (b) optional-dep gating (skip if `econml` absent)
  vs hard dep; (c) synthetic vs real data. **First step:** add extra + one example + smoke test.

### 4. Blocking stationarity / cointegration / overlap gates
- **What:** the `src/validation/` diagnostics currently **warn**; add an opt-in strict mode
  that **blocks** (CLAUDE.md: "may warn but should not block" — this is the future toggle).
- **Touches:** `src/validation/` diagnostics, an estimator-level `strict=`/`enforce=` flag,
  tests. **Effort:** Medium.
- **Sub-decisions:** (a) opt-in flag vs default-block; (b) which diagnostics block; (c)
  thresholds; (d) exception types. **First step:** plumb a `strict=` flag through one estimator.

### 5. Book content expansion (~180 → ~285 pages)
- **What:** per the existing plan `docs/plans/active/2026-03-01_22-00_content_expansion.md`.
- **Touches:** `chapters/*.tex` **+ parallel MDX port + W3 drift re-stamp** (the drift
  machinery now exists, so every expanded `.tex` needs its MDX kept in sync).
- **Effort:** Very large (content writing). **First step:** review that plan, pick next section.

## Recommended sequencing (my judgment — the user chooses)

1. **#2 namespace migration first** *if any new code is coming* — it's the flagged cleanup
   gate and avoids renaming new estimators later. Low methodology risk.
2. **#1 dynamic g-estimation** — highest-leverage methodology piece; closes the honesty gap
   the whole repo guards. Big, research-heavy.
3. **#3 EconML examples** — cheap, high-value, low-risk; good "warm-up" or parallel task.
4. **#4 blocking gates** — smaller hardening.
5. **#5 content expansion** — largest; has its own plan; pairs with the MDX port pipeline.

If the goal is *visible methodology credibility*, do #1. If *shippable public package*, do #2.
If *quick wins*, do #3.
